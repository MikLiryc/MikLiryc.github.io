GoMap GoTile 렌더 최적화

2024.07 ~ 2024.09 · Transformers Alliance (Unity)

발단

프로젝트의 Lobby 씬에서 GoMap을 활용하여 현재 위치 중심의 지도를 표시한다. 지도는 GoTile 오브젝트로 이루어진 타일들로 구성되며, 각 타일에는 해당 위치의 도로·강·건물에 해당하는 오브젝트가 생성된다. 화면에 렌더되는 타일 수가 많아질수록 퍼포먼스가 떨어지는 현상이 발생하였다.

현상 파악

봉은사역 부근 기준, 유니티 에디터에서 360도 회전하며 측정한 최소/최대 Batches 결과:

Batches SetPass calls CPU FPS
최소 Batches 1,333 267 main 8.4ms / render 6.0ms 118.5
최대 Batches 7,324 302 main 14.5ms / render 11.9ms 68.8

화면을 돌리기만 했을 뿐인데 Batches가 6,000개 가량 차이가 발생. 도로·강·빌딩 등 오브젝트가 밀집된 타일이 시야에 들어올 때 성능에 유의미한 영향을 주고 있었다.

접근 방법

Unity의 Batches(Draw Call)는 렌더되는 오브젝트의 mesh × material 수에 비례한다. 따라서 동일 Material을 사용하는 오브젝트들(도로끼리, 강끼리, 빌딩끼리)의 Mesh를 타일 단위로 하나로 병합하면, 수천 개의 Batches를 Material 종류 수만큼으로 줄일 수 있다고 판단하였다. 

public class CombineMesh : MonoBehaviour
{
    private Dictionary<string, CombineMeshData> _combineMeshDictionary = new();

    private void Start()
    {
        MeshFilter[] meshFilters = GetComponentsInChildren<MeshFilter>();
        MeshRenderer[] meshRenderers = GetComponentsInChildren<MeshRenderer>();

        // Material 별로 Dictionary에 분류
        ClassifyRenderers(meshFilters, meshRenderers);

        // Material별로 Mesh 병합 후 새 오브젝트 생성
        foreach (var combineMeshData in _combineMeshDictionary)
        {
            var go = new GameObject(combineMeshData.Value._sharedMaterial.name);
            go.transform.SetParent(transform);
            MeshFilter mf = go.AddComponent<MeshFilter>();
            MeshRenderer mr = go.AddComponent<MeshRenderer>();
            mr.sharedMaterial = combineMeshData.Value._sharedMaterial;
            mf.mesh = new Mesh();
            mf.mesh.indexFormat = UnityEngine.Rendering.IndexFormat.UInt32;
            mf.mesh.CombineMeshes(combineMeshData.Value._combineInstanceList.ToArray());
        }
    }
}

전투 시스템 파이프라인 (FSM + Timeline 연출)

2024.10 ~ 2026.02 · Transformers Alliance (Unity)

발단

프로젝트의 전투 시스템은 최초 단선형 코루틴 기반으로 구현되어 있었다. 인트로 연출 → 전투 → 결과까지의 흐름이 하나의 긴 코루틴 체인으로 이루어져 있어, 새로운 연출을 추가하거나 예외 상태(재전투, 중도 이탈 등)를 처리할 때마다 기존 코드 전반을 수정해야 했다. 특히 Cinemachine + Timeline을 활용한 카메라 연출이 점점 복잡해지면서, 상태 전환 시 카메라 위치가 한 프레임 튀는 문제와 재전투 루프에서의 프레임 드롭이 심각한 수준에 이르렀다.

현상 파악

  • 인트로 → 대치 → 스킬 사용 장면 전환 시, Timeline 전환 직후 1프레임 카메라 위치 튐 발생
  • 재전투 반복 시 이전 전투의 오브젝트/이벤트 리소스가 완전히 정리되지 않아 프레임 드롭 누적
  • 스킬 성공/실패/강스킬 분기가 if-else 체인으로 처리되어, 신규 스킬 타입 추가 시 20+ 개소 수정 필요
  • 전투 진입 UI(팀 편성, 포션, 시뮬레이션)와 전투 로직 간 강결합으로 독립 테스트 불가

접근 방법

전투 전체 흐름을 FSM(Finite State Machine) 아키텍처로 재설계하였다. 각 상태는 독립적인 클래스로 분리되어, 진입(Enter) → 실행(Execute) → 퇴장(Exit) 생명주기를 갖는다. 상태 간 전환은 BattleManager가 중앙에서 관리하며, 각 상태는 자신의 Timeline/Cinemachine 설정만 책임진다. 

// ── 전투 상태 머신 구조 (의사 코드) ──

public abstract class BattleState
{
    protected BattleManager _manager;
    public virtual void Enter() { }    // Timeline 바인딩, 카메라 세팅
    public virtual void Execute() { }  // 매 프레임 로직
    public virtual void Exit() { }    // 리소스 정리, 카메라 복원
}

// 상태 흐름:
// Intro(아군/적군/전체) → Confrontation(대치)
//   → SkillReady → [Success] StrongSkill / [Fail] SkillFail
//     → HealthCheck → [HP 잔여] Confrontation (반복)
//                   → [HP 0] Result(승/무/패)

public class BattleManager
{
    private BattleState _currentState;

    public void ChangeState(BattleState newState)
    {
        _currentState?.Exit();
        _currentState = newState;
        _currentState.Enter();
    }
}

게임 내 '통신' 시스템으로 사용되는 다이얼로그 시스템은 단순한 텍스트 출력을 넘어, 다양한 연출 요소를 텍스트와 동기화해야 하는 복잡한 요구사항을 가지고 있었다. 텍스트 출력 도중 배경음 전환, 효과음 재생, 카메라 셰이크, 캐릭터 애니메이션 등이 특정 글자 위치에서 실행되어야 하며, 대화 중 긴급 통신이 수신되면 현재 대화를 보존하고 복귀할 수 있어야 했다.

현상 파악

  • 기존 텍스트 출력은 단순 코루틴 기반으로, FPS에 따라 글자 출력 속도가 달라지는 문제 발생
  • BGM/SFX 등 연출을 텍스트와 동기화하려면 각 라인마다 별도의 하드코딩이 필요했음
  • 대화 A 진행 중 대화 B 진입 → 대화 B 종료 후 대화 A의 상태(텍스트 위치, 화자, 배경)가 유실
  • 기획자가 연출을 추가하려면 프로그래머에게 매번 코드 수정을 요청해야 하는 비효율

접근 방법

세 가지 핵심 모듈로 시스템을 설계하였다: 텍스트 엔진(DialogSequencer), 커맨드 파서(CommandParser), 상태 관리(Memento).

▸ 1. DialogSequencer — DeltaTime 보정 텍스트 엔진

// 글자 출력 타이밍을 DeltaTime으로 보정하여 FPS 독립적으로 동작
private float _elapsed;
private float _charInterval = 0.03f;

private void UpdateSequence()
{
    _elapsed += Time.deltaTime;
    while (_elapsed >= _charInterval && _currentIndex < _text.Length)
    {
        char c = _text[_currentIndex];
        if (c == '<')
        {
            // 커맨드 태그 감지 → CommandParser에 위임
            _currentIndex = _commandParser.Execute(_text, _currentIndex);
        }
        else
        {
            _displayText.Append(c);
            _currentIndex++;
        }
        _elapsed -= _charInterval;
    }
}

▸ 2. 인라인 커맨드 파서

기획자가 엑셀 테이블에서 직접 텍스트 안에 커맨드를 삽입할 수 있도록 설계하였다. 파서는 < > 태그를 만나면 커맨드를 추출하여 해당 시스템에 실행을 위임한다. 

// 지원 커맨드 예시 (엑셀 테이블에서 직접 작성 가능)
"적이 접근하고 있습니다!<CameraShake:Heavy> 경계를 <SFX:Alert01>유지하세요."
"<BGM:BattleTheme02>전투 준비를 시작합니다."
"<Animation:Salute>임무를 완수했습니다, 사령관."
"<Wait:1.5>잠시 후...<Wait:0.5>통신이 복구되었습니다."

// CommandParser 구조
public class CommandParser
{
    private Dictionary<string, ICommandHandler> _handlers;

    public int Execute(string text, int startIndex)
    {
        // <Tag:Param> 형식 파싱 → 핸들러 실행 → 닫힌 괄호 다음 인덱스 반환
        (string tag, string param, int endIndex) = ParseTag(text, startIndex);
        if (_handlers.TryGetValue(tag, out var handler))
            handler.Execute(param);
        return endIndex;
    }
}

▸ 3. Stack 기반 Memento 패턴 상태 관리

긴급 통신 수신 시 현재 대화 상태를 DialogMemento 객체로 캡처하여 스택에 Push한다. 긴급 통신 종료 후 Pop하여 이전 대화의 정확한 위치에서 복원한다. 

public class DialogMemento
{
    public int DialogGroupId;
    public int CurrentLineIndex;
    public int CurrentCharIndex;
    public string CurrentSpeaker;
    public string CurrentBGM;
    public ActorState[] ActorStates;  // 화자별 위치/표정 상태
}

public class DialogManager
{
    private Stack<DialogMemento> _stateStack = new();

    public void PushAndStartNew(int newDialogGroupId)
    {
        _stateStack.Push(CaptureCurrentState());   // 현재 상태 보존
        StartDialog(newDialogGroupId);               // 새 다이얼로그 시작
    }

    public void PopAndRestore()
    {
        if (_stateStack.Count > 0)
            RestoreState(_stateStack.Pop());          // 이전 상태 복원
    }
}

▸ 4. GraphView 다이얼로그 에디터

기획자의 작업 효율을 위해 Unity GraphView API 기반의 비주얼 다이얼로그 에디터를 개발하였다. DialogProfile, DialogBlock, DialogChoiceNode 등 노드 타입을 구현하고, 포트 연결 시 자동으로 데이터가 동기화된다. XlsxFileReader를 통해 엑셀 테이블 데이터를 읽어 노드로 자동 생성하는 기능도 포함하였다.

AR 소환게임 연출 시스템

2023.03 ~ 2025.12 · Transformers Alliance (Unity)

발단

소환게임은 LBS 맵에서 에너존을 발견한 뒤, 스파크를 물리적으로 발사하여 성공하면 AR 화면으로 진입하여 트랜스포머를 소환하는 핵심 콘텐츠이다. 프로토타입 단계에서는 단순 발사 → 결과 표시 흐름이었으나, 균열 탐색 → 미니게임 → AR 바닥 인식 → 포탈 생성 → 트랜스포머 소환까지의 전체 단계별 연출이 필요해졌고, 각 단계 간 자연스러운 전환과 카메라 각도에 따른 동적 배치가 핵심 과제가 되었다.

현상 파악

  • AR Foundation의 바닥 인식(Plane Detection) 정밀도가 낮아 포탈 위치가 부자연스럽게 배치되는 현상 발생
  • 카메라를 아래로 향하면 포탈이 너무 가깝고, 위로 향하면 너무 멀어져 연출 일관성이 깨지는 문제
  • 균열 탐색 → AR 진입 → 소환 완료까지의 플로우가 분리되어 있어 장면 간 끊김이 체감됨
  • 에디터에서 AR 테스트가 불가능하여 디바이스 빌드 없이는 연출 확인 불가

▸ 1. 스파크 물리 발사 메커닉

// 스파크 발사 각도·파워 제한, 물리 기반 궤적 계산
public class SparkLauncher
{
    private const float MIN_ANGLE = 45f;
    private const float MAX_ANGLE = 135f;
    private const float MIN_POWER = 0.3f;

    public void Launch(Vector2 direction, float power)
    {
        float angle = Mathf.Atan2(direction.y, direction.x) * Mathf.Rad2Deg;
        angle = Mathf.Clamp(angle, MIN_ANGLE, MAX_ANGLE);
        power = Mathf.Max(power, MIN_POWER);

        // 마찰력·가속도 적용한 궤적 시뮬레이션
        Vector3 velocity = CalculateVelocity(angle, power);
        _rigidbody.AddForce(velocity, ForceMode.Impulse);
    }
}

▸ 2. 카메라 각도 기반 포탈 위치 동적 보간

// 카메라 10~60도 사이에서 포탈 min~max 거리를 자동 보간
private Vector3 CalculatePortalPosition(Camera arCamera, Transform floorPlane)
{
    float camAngle = Vector3.Angle(arCamera.transform.forward, Vector3.down);
    float t = Mathf.InverseLerp(10f, 60f, camAngle);

    float distance = Mathf.Lerp(_minPortalDist, _maxPortalDist, t);

    // 카메라 정면 방향으로 바닥면에 투영된 위치 계산
    Vector3 forward = arCamera.transform.forward;
    forward.y = 0;
    forward.Normalize();

    return arCamera.transform.position + forward * distance;
}

▸ 3. 에디터 테스트 환경

실제 디바이스 없이도 AR 연출을 테스트할 수 있도록, 에디터 환경에서 임시 바닥 평면을 자동 생성하고 마우스 입력으로 카메라 각도를 조절하여 포탈 배치 및 소환 연출 전 과정을 확인할 수 있게 구현하였다.

  • 균열 탐색 시스템: 화면 기반 균열 위치 추적, 조준 성공 시 확장 연출 + 사운드/진동 피드백 연동
  • 쇼크웨이브 신호등: 성공/실패 확률 기반 3단계 신호등 연출, 타이밍에 따른 결과 분기
  • 속성별 차별화: 소환 속성(화/수/전기 등)에 따라 배경 환경, 이펙트, 포탈 색상을 다르게 적용
  • 도감 등록 연동: 소환 성공 시 트랜스포머 등장 타임라인 → 홀로그램 연출 → 도감 등록 플로우 자동 연결
